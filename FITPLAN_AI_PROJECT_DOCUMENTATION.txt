SECTION 1: PROBLEM STATEMENT
================================================================================

1.1 BACKGROUND
In the modern era, fitness and diet planning are crucial for maintaining good 
health. However, most people struggle with:
- Creating personalized fitness routines
- Following customized meal plans
- Tracking progress effectively
- Getting expert guidance without hiring expensive personal trainers

1.2 THE PROBLEM
Current fitness apps lack:

A. PERSONALIZATION ISSUES
   • Generic workout plans that don't consider individual health conditions
   • Meal plans that don't respect location-based food availability
   • No consideration for food preferences (vegetarian/non-vegetarian)
   • One-size-fits-all approach that doesn't work for diverse users

B. POOR USER ONBOARDING
   • Complex, multi-page forms that users don't complete
   • Unclear expectations about what data is needed
   • No guidance on fitness level or health conditions
   • Time-consuming setup process

C. LACK OF AI INTEGRATION
   • Static plans that don't adapt to user changes
   • No intelligent recommendations based on health data
   • Limited ability to generate varied meal plans
   • Manual plan creation with human limitations

D. ACCESSIBILITY PROBLEMS
   • Not mobile-friendly
   • Difficult to navigate
   • No quick reference for daily plans
   • Poor data visualization

1.3 IMPACT
Users waste time searching for generic fitness advice when they need:
   ✓ Personalized workout routines
   ✓ Region-specific meal plans
   ✓ Health condition awareness
   ✓ Easy-to-follow guidance
   ✓ Quantifiable progress tracking

================================================================================

2.1 FITPLAN.AI SOLUTION
Fitplan.ai is an AI-powered fitness and diet planning platform that:

A. INTELLIGENT PERSONALIZATION
   • Uses Google Gemini AI for plan generation
   • Considers location for regional cuisine
   • Respects food preferences (veg/non-veg)
   • Adapts to health conditions and fitness levels
   • Real-time BMI calculation with target weight

B. STREAMLINED ONBOARDING (5-STEP PROCESS)
   Step 1: Basic Information (Name, Email, Age, Gender)
   Step 2: Fitness Profile (Height, Weight, Level, Goal, BMI)
   Step 3: Health & Preferences (Location, Health Issues, Food Pref)
   Step 4: Confirmation (Review all details)
   Step 5: Generation (AI creates personalized plans)

C. AI-POWERED PLAN GENERATION
   • Workout Plans: Customized exercises based on fitness level and goal
   • Diet Plans: Location-aware, region-specific meals
   • Real-time adaptation: Change preferences → Regenerate plans
   • Specific quantities: Meals include exact ingredient amounts

D. USER-FRIENDLY DASHBOARD
   • Home View: Quick daily overview
   • Detailed Views: Complete meal and workout plans
   • Progress Tracking: Visual charts and metrics
   • Settings: Easy profile management
   • Editable Food Preferences: Change anytime

SECTION 4: SYSTEM ARCHITECTURE
================================================================================

4.1 COMPONENT ARCHITECTURE

Frontend Components:
  • Login/Registration: User authentication
  • Onboarding: 5-step setup wizard
  • Dashboard: Home, Diet, Workout, Progress, Settings
  • UI Components: 40+ reusable components

Backend Modules:
  • main.py: FastAPI app and routes
  • models.py: Pydantic data models
  • geminiapi.py: AI integration
  • database.py: MongoDB connection

Database Schema:
  users
    ├─ user_id (Primary Key)
    ├─ email (Unique Index)
    ├─ password_hash
    └─ created_at

  user_profiles
    ├─ profile_id
    ├─ user_id (Foreign Key Index)
    ├─ age, gender, height, weight
    ├─ fitness_level, goal
    ├─ location, health_issues
    ├─ food_preference ('veg' or 'mixed')
    └─ updated_at

  workout_plans
    ├─ plan_id
    ├─ user_id (Foreign Key Index)
    ├─ exercises (array)
    ├─ days, duration
    └─ created_at

  diet_plans
    ├─ plan_id
    ├─ user_id (Foreign Key Index)
    ├─ meals (array with quantities)
    ├─ location, food_preference
    ├─ total_calories, total_protein
    └─ created_at

4.2 DATA FLOW

1. USER REGISTRATION
   Input: Email, Password, Name
   ↓ (Validation & Hashing)
   Store: MongoDB users collection
   ↓ (Generate JWT)
   Output: JWT Token (24h expiration)

2. ONBOARDING
   Input: 5-step form data
   ↓ (Validation)
   Store: user_profiles collection
   ↓ (Trigger AI)
   Output: User profile + Plans cached

3. PLAN GENERATION
   Input: User Profile + Location + Preferences
   ↓ (Format Gemini Prompt)
   Call: Google Gemini 2.5 Flash API
   ↓ (Parse Response)
   Store: workout_plans, diet_plans collections
   ↓ (Cache Locally)
   Output: Plans in Zustand Store

4. PLAN DISPLAY
   Input: Plan from store
   ↓ (Format for UI)
   Display: Dashboard or Detail Page
   ↓ (User Interaction)
   Support: Update profile, regenerate, download

4.3 AUTHENTICATION FLOW

Registration:
  User → Email/Password → Backend → Hash (bcrypt) → Store → JWT → Frontend

Login:
  User → Email/Password → Backend → Verify Hash → JWT → Frontend

API Requests:
  Frontend → Include JWT → Backend → Verify Token → Validate → Response
  Token Valid? → Proceed | Token Expired? → Refresh | Invalid? → Reject

4.4 CACHING STRATEGY

Cache Layers:
  1. Browser localStorage: JWT token, user preferences
  2. Zustand Store: User profile, plans (session cache)
  3. MongoDB: Persistent storage
  4. CDN: Frontend assets (images, CSS, JS)

Cache Invalidation:
  • Plans: Invalidated when profile changes
  • Profile: Updated immediately on edit
  • Token: Refreshed automatically on expiration

4.5 SECURITY MEASURES

Frontend:
  ✓ Input validation before API calls
  ✓ XSS prevention (React escapes content)
  ✓ HTTPS only (on production)
  ✓ Secure token storage

Backend:
  ✓ Password hashing (bcrypt, 10 salt rounds)
  ✓ JWT validation on all protected endpoints
  ✓ Pydantic input validation
  ✓ SQL injection prevention (MongoDB parameterization)
  ✓ Rate limiting (future implementation)

Database:
  ✓ Authentication enabled
  ✓ User data isolation (by user_id)
  ✓ Regular backups
  ✓ Encryption at rest (MongoDB Atlas)

5.1 FEATURE 1: USER AUTHENTICATION & REGISTRATION

Implementation:
  Frontend:
    - Login.tsx: Simple email/password form
    - React Hook Form: Form validation
    - API call to /auth/register or /auth/login

  Backend:
    - POST /auth/register: Create user, hash password
    - POST /auth/login: Verify credentials, issue JWT
    - POST /auth/refresh: Refresh expired token

  Security:
    - Password hashed with bcrypt (never stored plain)
    - JWT tokens with 24-hour expiration
    - Tokens stored in localStorage (frontend)

Code Flow:
  User enters credentials → Client validation → API call → Backend validation
  → Password hash verification → JWT generated → Token sent to frontend
  → Stored in localStorage → Auth headers on future requests


5.2 FEATURE 2: ONBOARDING (5-STEP PROCESS)

Step-by-Step Implementation:

Step 1: BASIC INFORMATION
  Input Fields:
    • Name: Text input, 2-50 characters
    • Email: Auto-filled from registration
    • Age: Number input, 15-120
    • Gender: Select (Male/Female/Other)
  Validation: All required, age range check
  Storage: Form state in React

Step 2: FITNESS PROFILE
  Input Fields:
    • Height: Number (cm), 140-250
    • Weight: Number (kg), 20-300
    • Fitness Level: Select (Beginner/Intermediate/Advanced)
    • Goal: Select (Lose Weight/Gain Muscle/Maintain)
  Real-Time Calculation:
    • BMI = weight / (height in meters)²
    • Target weight for healthy BMI (22-24)
  Validation: All required, within ranges
  Display: Real-time feedback with color coding

Step 3: HEALTH & PREFERENCES
  Input Fields:
    • Location: Text input (city/region)
    • Health Issues: Multi-select checkboxes
      Options: Diabetes, High BP, Low BP, Heart Disease, Joint Issues,
               Back Pain, Others
    • Food Preference: Optional single button
      Option: "Vegetarian Only" (Default: Mixed)
  Storage: Arrays and flags
  Purpose: Personalize plans, respect preferences

Step 4: CONFIRMATION
  Display:
    • All entered information in organized cards
    • Edit buttons to go back to specific steps
    • Final confirmation button
  Validation: No new input, just review
  UX: Clear, organized presentation

Step 5: GENERATION
  Loading State:
    • Show "Generating Plans..." spinner
    • Disable buttons to prevent double-submission
  API Calls:
    • POST /ai/workout (with profile data)
    • POST /ai/diet (with profile + location + food preference)
  Processing:
    1. Send profile to backend
    2. Backend formats Gemini prompt
    3. Call Gemini API
    4. Parse structured response
    5. Save to MongoDB
    6. Return to frontend
    7. Cache in Zustand store
  Success:
    • Display success message
    • Navigate to dashboard
    • Plans ready for viewing

Data Structure:
  onboardingData = {
    step1: { name, email, age, gender },
    step2: { height, weight, fitnessLevel, goal, bmi, targetWeight },
    step3: { location, healthIssues[], foodPreference },
  }

State Management:
  Zustand Store:
    • setUserProfile(data) - Save complete profile
    • setWorkoutPlans(plans) - Cache workout
    • setDietPlans(plans) - Cache diet
    • clearAllPlans() - Reset for regeneration

5.4 FEATURE 4: LOCATION-AWARE DIET PLANNING

Key Innovation: Regional cuisine consideration

Location Examples:
  • India: Dal, Paneer, Rice, Roti, Sambar, Idli
  • USA: Chicken, Broccoli, Brown Rice, Fruits
  • Mediterranean: Olive Oil, Feta, Vegetables
  • East Asia: Rice, Tofu, Vegetables, Seafood

Gemini Prompt Format:
  "Generate a 7-day personalized diet plan:
   - User Location: [location]
   - Food Preference: [veg/mixed]
   - Calorie Target: [calculated based on goal]
   - Health Conditions: [list]
   
   Use local cuisine and ingredients from [location].
   Each meal must include:
   - Meal type (breakfast/lunch/dinner)
   - Time
   - Dish name
   - Description
   - Calories and macros (protein, carbs, fats)
   - Food type (veg/non-veg)
   
   CRITICAL FOR QUANTITIES:
   For each dish, list exact quantities:
   - Item: quantity unit (e.g., Idli: 3 pieces, Sambar: 1 cup)
   - Never approximate (not just '1 plate')
   
   Format as JSON with meals array."

Food Preference Handling:
  • Veg Only: Include only vegetarian meals
  • Mixed (default): Include both veg and non-veg
  • Passed to Gemini: Affects meal generation

Quantity Specification:
  Important: Each ingredient has:
    • Item name
    • Exact amount (number)
    • Unit (pieces, cups, grams, tbsp, ml, etc.)
  Example:
    "Idli: 3 pieces"
    "Sambar: 1 cup"
    "Chutney: 2 tbsp"

Output Structure:
  {
    "mealPlan": [
      {
        "day": 1,
        "meals": [
          {
            "mealType": "breakfast",
            "time": "8:00 AM",
            "name": "Idli with Sambar",
            "description": "South Indian specialty",
            "calories": 250,
            "protein": 12,
            "carbs": 40,
            "fats": 5,
            "foodType": "veg",
            "quantity": [
              { "item": "Idli", "amount": "3", "unit": "pieces" },
              { "item": "Sambar", "amount": "1", "unit": "cup" }
            ]
          },
          ...
        ]
      },
      ...
    ]
  }

Display:
  • DashboardHome: Compact view (meal names only)
  • DietPlanPage: Detailed view with quantities
  • Quantity section: Green highlight for easy identification

    • Instruction: "User food preference: [veg/mixed]"

5.6 FEATURE 6: REAL-TIME BMI CALCULATOR

Formula:
  BMI = weight (kg) / (height in meters)²

Implementation:
  Location: Onboarding Step 2
  
  Triggers:
    • When height input changes
    • When weight input changes
    • On form load (if values exist)
  
  Calculation:
    const heightInMeters = height / 100;
    const bmi = weight / (heightInMeters * heightInMeters);
    return bmi.toFixed(1);

Display:
  • Real-time update as user types
  • Show BMI value: "26.2"
  • Color-coded status:
    < 18.5: Yellow (Underweight) ⚠️
    18.5-24.9: Green (Normal) ✓
    25-29.9: Orange (Overweight) ⚠️
    >= 30: Red (Obese) ❌

Target Weight Suggestion:
  Formula: Aim for BMI 22 (middle of healthy range)
  targetWeight = 22 × (height in meters)²
  
  Display:
    "Your current BMI: 26.2 (Overweight)"
    "Target weight for healthy BMI: ~71 kg"
    "Weight to lose: ~14 kg"

Storage:
  • Calculated but not stored permanently
  • Recalculated on profile update
  • Used for recommendations

6.6 ERROR HANDLING WORKFLOW

API Error Workflow:
  User action triggers API call
    ↓
  Frontend validation passes
    ↓
  Request sent to backend
    ↓
  Server processes request
    ↓
  Error occurs (401, 400, 500, etc.)
    ↓
  Error response sent
    ↓
  Frontend receives error
    ↓
  Check error type:
    • 401 Unauthorized: Token expired? Refresh or re-login
    • 400 Bad Request: Show validation error message
    • 500 Server Error: Show "Please try again later"
    • Network Error: Show "Check internet connection"
    ↓
  Display toast notification
    ↓
  User can retry action


Token Expiration Workflow:
  Token expires after 24 hours
    ↓
  Next API call includes expired token
    ↓
  Backend returns 401 Error
    ↓
  Frontend catches 401
    ↓
  Attempt token refresh: POST /auth/refresh
    ↓
  New token received
    ↓
  Retry original API call
    ↓
  Proceed normally
    
  If refresh fails:
    ↓
  Redirect to login
    ↓
  User must re-authenticate
